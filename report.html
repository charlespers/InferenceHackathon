Decisive B=1 finding (K6 measured): TP=8 sharding runs each GPU at 118 GB/s (3.5% HBM peak) vs single-GPU 693 GB/s (20.7%) — the ~8× data reduction is cancelled by ~6× worse per-GPU efficiency (slices too small to saturate an H100). TP=8 does NOT help B=1 latency. The 8-GPU path is EP (Expert Parallelism): 94 all-to-all collectives (1 NVLink barrier each, NVSHMEM put-barrier 17µs) instead of 188 TP all-reduces (3 barriers × ~17µs = 51µs). EP + batched spec: 94 × 17µs = 1.6ms ÷ 2.77 + ~0.5ms compute ≈ 1.1ms → ~960 tok/s.
Apples-to-apples: custom EP+spec ~960 (projected) vs vLLM 65.8–85.7 (measured) — ~11×. The 1-GPU fused step (31.9) and TP=8 sharded (33.8) are measured development milestones, not the deployment config. The EP-sharded decode needs wiring end-to-end.

The path to 960+: EP=8 (assign 16 experts/GPU, each GPU handles full expert ops — no occupancy starvation) → 94 NVSHMEM all-to-all collectives (1 barrier × 17µs each vs TP's 3 barriers × ~17µs = 51µs) → batched speculative decode (EAGLE3 ÷2.77–3.8, amortizes both compute and comms).
